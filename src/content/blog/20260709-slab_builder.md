<h2 id="part-1專案動機">Part 1：專案動機</h2>

在進行計算化學的表面模擬時，經常需要頻繁地從塊材（Bulk）結構中切出特定的晶面（Slab）並加上真空層。傳統上，這需要手動修改參數、打指令，再丟進外部軟體（如 VESTA）檢查結構，過程繁瑣且容易出錯。

與其每次都重複這些機械化的動作，我決定化被動為主動，開發一款自動化的小工具。目標是打造一款**無須依賴外部軟體、內建 3D 視覺化、且能自動輸出 VASP POSCAR 格式**的桌面端應用程式，藉此提升未來在實驗室處理計算前置作業的效率。

<h2 id="part-2建立過程與對話紀錄-prompt--response">Part 2：建立過程與對話紀錄 (Prompt & Response)</h2>

這個專案的開發過程並非一蹴可幾，而是透過與 AI 不斷地對話、測試與修正才成型。以下記錄了軟體的進化史，以及我在每個階段的關鍵提問（Prompt）與 AI 的解決方案。

### Phase 2.1：Python 腳本起步與環境建置
一開始，我需要先確認核心的運算邏輯與開發環境。
* 🗣️ **我的 Prompt：** 
  > 「這份檔案是我們老師所出的作業，幫我整理一下看他需要什麼內容... 那可以安排一下順序嗎？我現在要開始我的第一步是什麼？那幫我從第一步開始，一步一步帶著我做。」
* 🤖 **AI 的回覆與行動：**
  AI 協助梳理了作業要求，並指導我在 Mac 上建置 VS Code 與 Python 3 環境。接著，我們使用了 **ASE (Atomic Simulation Environment)** 套件，成功寫出第一支能在終端機裡讀取 `POSCAR` 並利用 `ase.build.surface` 切出 Pt(111) 晶面的 Python 腳本。
![slab](/blog/20260709/20260709-1.png)

### Phase 2.2：邁向圖形化 — 建置 Streamlit 網頁應用程式
腳本測試成功後，我們進入了作業要求的「工具包裝」階段，最初的計畫是做成網頁介面。
* 🤖 **AI 的回覆與行動：**
  AI 帶領我使用 `Streamlit` 框架，將原本的終端機腳本升級為具備側邊欄參數輸入、上傳與下載功能的網頁。我們同時規劃了專業的專案資料夾結構（包含 `examples` 與 `outputs`），建立 `requirements.txt`，並成功在本地端跑出網頁，準備測試老師規定的 Pt(111)、Pt(100) 與 Cu(111) 三個案例。
![slab](/blog/20260709/20260709-2.png)
![slab](/blog/20260709/20260709-3.png)


### Phase 2.3：遭遇網路瓶頸，果斷轉向桌面端 App (Tkinter)
雖然 Streamlit 網頁初步建置成功，但我在執行時遭遇了本機網路連線超時（Timeout）的狀況，導致網頁卡頓。為了追求更高的軟體穩定性，我決定改變工具型態。
![slab](/blog/20260709/20260709prompt-1.png)
* 🗣️ **我的 Prompt：** 
  > 「我的網頁顯示連線超時，會不會有機會做成 app 的形式？」
* 🤖 **AI 的回覆與行動：**
  > 「這是個非常棒的點子！我們可以直接使用 Python 內建的視窗介面套件 `tkinter`，幫你做一個真正在 Mac 桌面上彈出來的軟體視窗。」
  AI 隨即改寫了程式架構，徹底捨棄網頁伺服器，成功打造出具備原生「選擇檔案」與「存檔」對話框的本機 App，完全避開了網路連線的風險。
這邊先產生的程式與網站一樣無法只能存取出POSCAR的檔案，需要再丟到VESTA上面做修正，所以後續再下一個問題中繼續修正。

### Phase 2.4：挑戰進階功能 — App 內建 3D 視覺化
傳統作業流程需要將產生好的 POSCAR 檔案丟進外部軟體 VESTA 才能看圖，我希望能將兩者合而為一。
![slab](/blog/20260709/20260709prompt-2.png)
* 🗣️ **我的 Prompt：** 
  > 「有沒有機會不透過 VESTA，讓我產生一個連透視圖都在 app 上的程式？」
* 🤖 **AI 的回覆與行動：**
  > 「這個想法太棒了！我們可以使用 Python 的繪圖套件 `matplotlib`，在視窗右半邊直接挖出一塊 3D 繪圖區。」
  AI 透過 `FigureCanvasTkAgg` 將 Matplotlib 無縫嵌入 Tkinter 介面中，抓取 ASE 產生的三維座標，成功畫出可拖曳旋轉的立體原子散佈圖。
![slab](/blog/20260709/20260709-4.png)
生成出的透視圖，看起來很粗糙，以及使用起來不順，在下一個問題中進行優化。

初步的 3D 圖形比例嚴重失真，且視覺背景雜亂。我直接提供了 VESTA 的截圖作為參考基準，要求優化視覺質感。
![slab](/blog/20260709/20260709prompt-3.png)
這邊我先提出的問題，是優化一下，但Gemini未能理解我想要優化的方向，所以就使用了下面這張圖的使用方式。
我給他VESTA生成出來的透視圖形式，讓他知道我想要他去更改的方向。

![slab](/blog/20260709/20260709prompt-4.png)
* 🗣️ **我的 Prompt：** 
  > 「轉起來很不順，而且顯示起來很奇怪，可以幫我優化一下嗎？... 比較一下，幫我做成白色球那張圖的樣子，那張圖是來自 VESTA。」
* 🤖 **AI 的回覆與行動：**
  AI 加入了 `ax.set_box_aspect()` 強制鎖定真實物理比例解決變形問題；同時將原子的顏色改為 VESTA 風格的銀白色 (`#F0F0F0`)，並關閉了所有醜陋的預設灰色背景與格線 (`ax.set_axis_off()`)，視覺質感大幅躍升。

到此已經完成了大致的軟體設計，不確定還需要再修正的地方，所以決定與老師討論。

### Phase 2.6：導入導師回饋與科學細節
在與指導老師討論後，我帶回了三個專業的修改建議，並明確給予重要性排序，確保程式碼邏輯符合科學嚴謹度。
![slab](/blog/20260709/20260709prompt-5.png)
* 🗣️ **我的 Prompt：** 
  > 「問完老師了，以下是幾個我需要去修改的點：1. 把真空層的空間標示出來 2. 在左下角顯示出一個 xyz 軸 3. 新增顏色在上面，因為以後可能會遇到各種不同的金屬... **123 就是這個重要性的程度**，幫我目前的程式來做一下修正。」
* 🤖 **AI 的回覆與行動：**
  AI 完全遵照優先級：寫迴圈畫出 Unit Cell 的 12 條外框線標示真空層、在原點畫出紅綠藍三色的 `quiver` 箭頭作為座標軸，並導入了 `ase.data.colors` (Jmol 官方演色表)，讓 Pt、Cu 等金屬能自動顯示標準化學顏色。

為了解決無法同時比較多個結構的痛點，我提出了多分頁與連動座標的構想。

* 🗣️ **我的 Prompt：**

<h2 id="part-3開發過程遇到的-bug-與解決方式">Part 3：開發過程遇到的 Bug 與解決方式</h2>

在開發過程中，踩過了不少坑，也累積了寶貴的除錯經驗：

1. **環境建置障礙 (`zsh: command not found: pip`)**
   * **問題：** Mac 終端機不認得 `pip` 安裝指令。
   * **解決：** 釐清 Mac 預設環境後，改用 `pip3 install` 或 `python3 -m pip` 成功喚醒安裝程式。
2. **格式讀取錯誤 (`RuntimeError: The number of scaling factors...`)**
   * **問題：** 程式崩潰，無法讀取 POSCAR。
   * **解決：** 發現是「人為失誤」。一是 VS Code 檔案忘記存檔（檔名旁有白點），二是 VASP 的 POSCAR 格式極度嚴格，不能有多餘的空白行或漏掉註解行。
3. **網頁端連線超時 (Streamlit Timeout)**
   * **問題：** 在啟動 Streamlit 網頁時，終端機卡住並報錯連線逾時。
   * **解決：** 研判可能是本機防火牆或埠號衝突，果斷改變軟體型態，將工具重構成不需要依賴瀏覽器與網路伺服器的 Tkinter 桌面端應用程式。
4. **3D 模型比例失真變形**
   * **問題：** 因為加入了很長的真空層，Matplotlib 預設將繪圖區壓縮成正方體，導致 XY 軸被嚴重擠壓。
   * **解決：** 透過程式抓取結構的真實物理長度，並加上 `ax.set_box_aspect()` 強制鎖定長寬高比例。
5. **XYZ 座標軸無法跟隨旋轉**
   * **問題：** 左下角的小座標軸與主畫面是兩塊不同的畫布，旋轉主畫面時座標軸會死當在原地。
   * **解決：** 撰寫 `motion_notify_event` 監聽器，讓兩塊畫布的 `elev` (仰角) 與 `azim` (方位角) 保持參數同步。

<h2 id="part-4未來的開發性與延伸性">Part 4：未來的開發性與延伸性</h2>

這個軟體目前已經具備了很高的完整度，但作為一個研究工具，還有許多可以持續擴充的方向：

1. **自動化吸附物測試：**
   目前的工具專注於切出「乾淨表面」。未來希望能加入自動放置吸附物（如 H、O、CO 分子）的功能，讓使用者能一鍵將分子精準放置於 Top, Bridge 或 Hollow site，直接產出能進行催化反應模擬的模型。
2. **進階表面分析與 AI 整合：**
   結合未來預計在本地端部署的開源 AI 模型（如 Ollama），或許能讓軟體具備初步判斷「最穩定晶面」或「推薦厚度」的輔助建議功能，打造更智能的計算化學工作站。

<h2 id="part-5結論">Part 5：結論</h2>

這兩天的開發是一場非常過癮的實作體驗。從一開始只會在終端機打指令、看報錯，到初步建立網頁版遇到網路挫折，最後成功融合了 ASE 運算核心、Tkinter 介面設計與 Matplotlib 3D 繪圖，打造出這款一體成型的「Slab Builder Pro」。

這不僅解決了當下的作業需求，更讓我體會到**「遇到痛點 ➔ 構思解法 ➔ 拆解步驟 ➔ 精準提問 (Prompting) ➔ 實作工具」**的工程師思維。這個軟體就像是在程式碼世界裡做出來的第一個完美晶體，期待它能成為我未來在化學領域探索時的得力助手。
